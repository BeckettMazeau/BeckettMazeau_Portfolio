Spring 2024

Java Image Processing Application

A custom-built photo editor written entirely in Java — featuring a pixel-manipulation engine, interactive Swing GUI, and a full suite of color filters, computer-vision effects, and geometric transformations engineered from the ground up.

Overview

The Challenge

Most image editing tools are black boxes — you apply a filter and something happens, but the underlying pixel math is hidden behind library calls. The goal of this project was to break that abstraction entirely: build a functional photo editor in Java where every filter, transformation, and color operation is implemented from first principles at the individual pixel level.

Rather than importing image processing libraries, the application models each pixel as a custom Color object carrying raw RGB data and color arithmetic methods. A 2D array of these objects forms a ColorImage — the core data structure that every filter, crop, and rotation operates on directly. This architecture made the relationship between algorithm and visual output concrete and immediately testable.

The full filter suite — sepia, monochrome, blur, contrast, edge detection, chroma keying, color-average smearing, and several geometric transforms — was designed to be accessible through an interactive Swing GUI, requiring no prior knowledge of image processing from the user.

Process

Design & Implementation

01

Pixel-Level Architecture

Rather than delegating color operations to a standard library, the project introduces Color.java — a custom RGB wrapper class that owns color data and exposes arithmetic for channel clamping, weighted blending, and distance calculation. This made every subsequent algorithm readable as a direct expression of the underlying color math rather than a series of bit-shifting operations on raw integers.

ColorImage.java builds on this by representing an entire image as a 2D Color array. Every filter method lives here, iterating over the grid and writing new color values based only on the class's own pixel data and the rules of the specific algorithm — no external state, no side effects.

02

Color & Artistic Filters

The foundational filter set covers the most common photographic adjustments. Sepia applies a classic warm tone through a fixed RGB weighting formula on each pixel. Monochrome averages the three channels to produce true grayscale. Contrast accepts a user-defined factor and scales each channel relative to a midpoint, with clamping to prevent overflow artifacts. The 3×3 box blur replaces every pixel with the mean of itself and its eight cardinal and diagonal neighbors — a foundational convolution operation implemented without any matrix library.

03

Advanced Computer Vision

Edge detection uses a color-distance formula to compare each pixel against its neighbors — when the distance exceeds a threshold, the pixel is classified as an edge. The filter offers two output modes: white edges on a black field, or colored edges composited over the original image. Both modes required careful handling of border pixels where the full 3×3 neighborhood is unavailable.

Chroma keying accepts a target background color and user-defined tolerance as R,G,B,T input. Any source pixel within that tolerance range is replaced with the corresponding pixel from a second image — enabling green-screen style compositing driven entirely by the custom color-distance logic in Color. The Color Average Shift effect creates a directional smear by computing a weighted blend of a pixel and a neighbor in a specified direction, producing a motion-blur-like artifact.

04

Geometric Transformations & I/O

Rotation and mirroring operate by constructing a new ColorImage with remapped dimensions and populating it by reading source pixel coordinates through a transformation function — keeping the original image immutable and making the coordinate math explicit. Cropping trims the 2D array by the specified top, bottom, right, and left pixel offsets, producing a new image of the reduced dimensions.

PhotoshopDriver.java handles all file I/O, converting between Java's BufferedImage and the internal ColorImage format on load and save. This boundary layer keeps image codec concerns entirely outside the processing engine, and means any JPEG-compatible image in the ./Images/ directory is immediately accessible from the GUI dropdown.

05

Swing GUI & User Workflow

GUI.java manages the complete Swing interface and event-listener layer. The workflow is deliberately linear: choose a source image from the left dropdown, select a filter from the main dropdown, supply any required parameters in the text field that conditionally appears, hit Apply Edit, and optionally save the result under a new filename. Parameter fields display inline formatting hints — so the user always knows whether to enter a single float, an R,G,B,T tuple, or crop offsets — without needing to consult documentation.

Reflection

Key Takeaways

01

Abstraction Through Custom Types

Wrapping raw RGB integers in a dedicated Color class paid dividends throughout the project. Every algorithm could express its logic in terms of colors and distances rather than bitwise arithmetic, keeping the code readable and making bugs traceable to specific color operations rather than silent integer overflow.

02

Immutability Simplifies Filters

Designing filter methods to produce new ColorImage instances rather than mutating in place eliminated an entire class of bugs where a filter's output pixel would contaminate the input for a neighboring pixel in the same pass — a subtle issue that would have been especially damaging in the blur and smear effects.

03

UI Feedback Reduces User Error

Conditional parameter fields that appear only when a filter requires input — and that show inline format hints — dramatically reduced invalid entries during testing. Coupling UI state directly to the selected filter in the event listener made the interaction model self-documenting rather than relying on a separate instruction panel.
