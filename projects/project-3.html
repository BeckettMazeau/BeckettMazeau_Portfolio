Java Ray Tracer

A physically-based ray tracer written entirely in Java — featuring Phong shading, mirror reflections, anti-aliasing, accurate shadow casting, and multi-threaded rendering, all driven by an interactive Swing GUI for building and exporting 3D scenes.

Overview

The Challenge

Ray tracing is one of the most direct ways to understand how light physically behaves — but implementing it from scratch means confronting every assumption that production renderers quietly paper over. The goal of this project was to build a complete, working ray tracer in Java where every lighting model, shadow, reflection, and material response is implemented from first principles.

Rather than relying on any graphics library, the renderer casts rays per-pixel, evaluates intersections against scene geometry, and computes color through the full Phong reflection model — including diffuse, specular, and mirror-reflected contributions. A custom multi-threaded architecture divides the image into horizontal chunks distributed across CPU threads, keeping render times practical even at high resolutions.

The accompanying Swing GUI makes the system fully interactive: users can compose scenes from spheres, planes, triangles, and polygons; adjust material and lighting properties; navigate a live low-resolution preview with a 6-DOF camera; and save full-quality renders to disk — all without touching code.

Process

Design & Implementation

01

Ray–Scene Intersection Engine

The core of the renderer is a per-pixel ray casting loop. For each pixel, a viewing ray is constructed from the camera origin through the pixel's position on the image plane. That ray is then tested against every object in the scene, and the closest intersection is selected as the visible surface point. From there, the material model at that point determines the returned color.

Intersection logic is implemented separately for each geometry type — spheres use the quadratic ray–sphere formula, planes use the dot-product formulation, and triangles and polygons use barycentric coordinate tests. This keeps each primitive's math self-contained and independently testable.

02

Lighting & Material Models

Four material types are supported. Lambert computes flat diffuse shading based on the angle between the surface normal and each light direction — simple, fast, and matte. Phong extends this with a specular highlight term that varies with the half-vector between the light and viewing directions, producing the characteristic shiny spot on curved surfaces. Mirror Phong adds a reflected-ray contribution, blending the Phong response with the color of whatever the reflected ray strikes. Perfect Mirror discards the material's own color entirely and returns only the reflected ray's color, producing a true mirror surface.

03

Shadows & Reflections

Shadow testing is performed by casting a secondary ray from the surface intersection point toward each light source. If any scene object lies between that point and the light, the light's contribution is excluded from the color calculation. This is repeated per light, so scenes with multiple lights can produce overlapping shadow regions correctly.

Reflections follow the same mechanism: at a reflective surface, a new ray is constructed as the mirror of the incoming viewing ray about the surface normal, and the renderer evaluates what that ray hits. The returned color is then composited into the current pixel's result according to the material's reflectance weight.

04

Anti-aliasing

A single ray per pixel can clip just past the edge of an object, producing jagged staircase artifacts — especially visible on curved surfaces and thin geometry. Anti-aliasing addresses this by subdividing each pixel into an N×N grid of sub-pixels, casting one ray per sub-pixel, and averaging the resulting colors into the final pixel value. The sample count is exposed in the Settings panel, so the user can trade off quality against render time. Even a 3×3 sample grid (9 rays per pixel) substantially smooths curved edges at moderate cost.

05

Multi-threaded Rendering

Rendering a full-resolution image is computationally expensive — each pixel may spawn multiple rays for shadows, reflections, and anti-aliasing samples. To reduce wall-clock time, the image is partitioned into equal horizontal strips and each strip is handed to a dedicated Runnable that processes its assigned rows independently. Each Runnable is wrapped in a Thread and started simultaneously. Because each thread writes to a distinct region of the shared ColorImage, no synchronization is needed during the render pass. The main thread then calls join() on all worker threads before writing the output file, ensuring a complete image.

06

Interactive GUI & Scene Workflow

The GUI is organized across three panels navigable from a top dropdown. The Settings panel configures output resolution, aspect ratio, FOV, anti-aliasing, and threading before any scene work begins. The Scene Creator panel lets users add objects to an existing or new scene — selecting object type, setting a name, and clicking Create New Object. Update Mode opens a secondary tab for modifying or removing objects by name; all objects sharing a name are updated together, making named objects the recommended workflow. The Viewer panel provides a live low-quality preview navigable by keyboard, so users can frame their shot before committing to a full-resolution save.

Reflection

Key Takeaways

01

Physics First, Abstraction Second

Implementing lighting and reflection from the underlying geometry — rather than calling library routines — made every visual artifact immediately traceable to a specific formula. When the Eye of Sauron scene produced unexpected color banding, the source was identifiable as a missing normalization step in the specular term, not a mystery inside a black-box shader.

02

Concurrency Requires Careful Data Layout

Multi-threaded rendering is straightforward when threads write to non-overlapping memory regions. Partitioning the image by row meant threads never contended over the same pixel, eliminating the need for locks and keeping the parallel speedup close to linear with core count. Designing data structures for thread safety from the start avoided retrofitting synchronization later.

03

Preview Quality as a UX Trade-off

Rendering the viewer preview at reduced resolution keeps the GUI responsive during camera navigation while still giving an accurate composition preview. Separating preview quality from save quality — and communicating that separation clearly in the UI — let the system feel fast without misleading users about final output.
